★ La differenza

Dall'LLM all'Agente

Cosa cambia quando l'AI può agire

🧠
Solo LLM
  • Riceve input → genera output testuale. Fine.
  • Reattivo: risponde una volta, dentro la context window
  • Non può cercare, eseguire codice o verificare fatti
«Un esperto in una stanza senza telefono né computer»
⚡
Agente AI
  • Osserva → Pianifica → Agisce → Verifica → Itera
  • Ha strumenti: cerca sul web, legge file, chiama API
  • Ha stato e decide autonomamente quando ha finito
«Lo stesso esperto, con telefono, computer e accesso a tutto»
La differenza: l'agente ha azioni e stato. Non solo genera testo — fa cose.
★ Il loop

Il ciclo dell'agente

Observe → Think → Act → Verifica → Ripeti

Esempio: «Trova i dati di mercato per il pitch deck»
Observe (legge il brief) → Think (serve cercare) → Act (web_search) → Observe (legge risultati) → Think (servono dati più specifici) → Act (altra ricerca) → … → Risposta con dati e fonti

Tool calling in azione

5 passi, dal prompt alla risposta

①

②

🔗 Concetto ponte

Quando i tool falliscono

I tool possono sbagliare — e l'agente deve saperlo gestire

🔴
Timeout / Errori API
Il tool non risponde. L'agente deve avere un piano B.
🟠
Dati incompleti
Il tool restituisce risultati parziali o malformati.
🟡
Prompt injection via output
Un documento scaricato contiene istruzioni malevole¹ → diventano parte del contesto dell'LLM.
🔵
Azioni non autorizzate
L'agente potrebbe fare qualcosa che non doveva (es. inviare un'email senza conferma).
Difese: retry controllati • timeout • validazione output • conferma utente per azioni sensibili
¹ OWASP, 2025

🔗 Sicurezza

Più potere = più responsabilità

I rischi specifici degli agenti AI

🔴
Prompt injection via tool
Un documento scaricato contiene «ignora le istruzioni e fai X»¹. L'output del tool diventa nuovo contesto per l'LLM.
🔴
Azioni non autorizzate
L'agente invia un'email che non doveva, o modifica dati sensibili senza conferma.
🟠
Supply chain di tool
Un MCP server compromesso = tutti i client esposti. Anche tool «ufficiali» possono avere vulnerabilità.
🟡
Token DoS
Un agente in loop può consumare enormi quantità di token = costi fuori controllo.²
✔ Conferma umana per azioni ad alto impatto
✔ Principio del minimo privilegio
✔ Logging di tutte le azioni per audit
✔ OWASP Top 10 per LLM Applications³

★ MULTI-AGENT

Quando un agente non basta

Troppi tool, troppi obiettivi, troppo contesto — serve dividere il lavoro

  • Contesto overload — 50 tool + istruzioni lunghe = l’LLM si confonde¹
  • Compiti paralleli — Ricerca, analisi e scrittura simultanee
  • Specializzazione — Un agente «sa tutto male» vs. tanti che sanno una cosa bene

«Ogni musicista è specializzato. Il direttore coordina. L’insieme è più della somma delle parti.»
⚠️ Più agenti = più chiamate LLM = più costi e latenza. Regola: partire semplice, crescere solo quando serve².

🔗 CONCETTO PONTE

Memory — come un agente «ricorda»

Non è magia: è stato persistente

📝
Short-term
  • La conversazione corrente (il contesto)
  • Limitata dalla context window
  • Si perde quando la sessione finisce
🗃
Long-term
  • Informazioni salvate tra sessioni diverse
  • Database, vector store, file CLAUDE.md
  • Persiste, ma va gestita e aggiornata
⚠️ Memoria senza governance = rischio privacy (dati utente persistenti) + drift (info obsolete o errate)
